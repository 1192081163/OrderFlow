# 本地单机邮件工作站设计

**日期：** 2026-07-13

**状态：** 已确认，待实施计划
**取代：** `2026-07-10-office-mail-gateway-design.md` 及对应云端网关实施计划

## 目标

把订单邮件读取、最近邮件缓存、新邮件通知、附件下载和订单提取全部放在一台办公室 Windows 电脑上运行。用户只登录企业邮箱，不再配置远程 API、服务器 token 或网关 Agent。程序不开放任何入站端口，邮件凭据和订单数据不离开办公室电脑。

## 已确认的使用方式

- 只支持一台办公室 Windows 电脑，不再向其他电脑共享邮件列表或通知。
- 首次使用时登录企业微信邮箱：邮箱地址加客户端授权码，连接 `imap.exmail.qq.com:993`。
- Windows 用户登录后程序自动启动；关闭主窗口时缩到系统托盘，只有托盘中的“退出”才停止程序。
- 后台持续监听新邮件并弹出 Windows 系统通知。
- 固定缓存最近 7 天邮件，不提供 3、14、30 天等时间范围切换。
- 只显示并通知含有效订单 Excel 附件的邮件；普通邮件不进入订单列表，也不通知。
- 收到邮件后不自动提取。用户仍在列表中手动勾选邮件，再点击提取。
- 提取继续使用现有本地 Python 规则、结果工作簿和输出目录按钮。

## 非目标

- 不建设或部署 `orderflow.ausmet.ai` 邮件 broker。
- 不提供多电脑同步、浏览器访问、远程控制或公网访问。
- 不安装登录前运行的 Windows Service。
- 不开放 HTTP、WebSocket、SOCKS、文件共享或其他监听端口。
- 不重写订单提取规则；`extract.py` 和现有 Python bridge 仍是规则源。
- 不改变现有本地文件/文件夹提取功能和输出目录交互。

## 方案选择

采用单个 Electron 应用内的本地工作站架构。Electron 主进程拥有邮箱连接、凭据、缓存、后台监听、通知和提取编排；渲染进程只通过 IPC 获取无敏感信息的视图和执行用户动作。

没有采用独立 Windows Service，因为本版本只要求用户登录后运行，额外服务会增加安装、权限和升级复杂度。没有采用“仅打开窗口时轮询”，因为该方式无法在缩到托盘后持续通知。

## 组件边界

### 1. 本地凭据存储

`LocalMailCredentialStore` 由 Electron 主进程创建，并通过注入的 `safeStorage` 操作实现：

- 邮箱地址和非敏感偏好写入本地 JSON。
- 授权码使用 Electron `safeStorage.encryptString` 加密；Windows 上由 DPAPI 保护。
- 渲染进程只能读取 `email`、`hasAuthCode`、`startAtLogin` 和连接状态，永远拿不到授权码。
- 如果 `safeStorage.isEncryptionAvailable()` 为假，程序可以测试本次登录，但不保存授权码，并明确要求下次重新登录；禁止回退到明文保存。
- 日志不得包含授权码、Authorization 头、完整附件内容或解密后的设置对象。

现有 `~/.order_organizer_assistant/email_settings.json` 如果含有明文 `authCode`，新版本首次启动时执行一次迁移：成功加密后重写 JSON，只保留邮箱地址和非敏感偏好。迁移失败时保留原文件并提示用户重新登录，不静默删除可恢复信息。

### 2. 本地邮件缓存

使用 Electron 39 自带 Node 22 的 `node:sqlite`，不增加原生 SQLite npm 包。已验证当前 Electron 39.8.10 可加载 `node:sqlite`。

数据库位于应用本地数据目录，启用 WAL。核心数据为：

- `mail_messages`：以 `mailbox_id + uid` 唯一标识，保存主题、发件人、日期、订单附件名称、首次发现时间、通知时间和提取状态。
- `mailbox_state`：保存当前邮箱、最近同步时间、最近 UID 和连接状态恢复所需的游标。

缓存只保留最近 7 天。同步重复执行时按邮箱和 UID 幂等更新，不重复插入、不重复通知。数据库不保存授权码或附件二进制。损坏时先将数据库改名备份，再创建新数据库并重新同步最近 7 天。

### 3. 邮箱监控器

`LocalMailboxMonitor` 只在主进程运行，并维护一条 IMAP TLS 连接：

- 优先使用 IMAP IDLE 接收邮箱变化事件。
- 每 60 秒执行一次轻量增量扫描，作为 IDLE 漏事件和服务器断连的兜底。
- Windows 从睡眠恢复、网络恢复或用户点击“重新连接邮箱”时立即执行增量扫描。
- 临时网络错误使用带抖动的 1–60 秒指数退避。
- 授权失败进入 `attention_required`，停止无意义重试，直到用户保存新的授权码。
- 初次登录和丢失游标时扫描最近 7 天；正常情况下从最后 UID 增量同步。

监控器先检查附件扩展名，再对候选 `.xlsx`/`.xlsm` 附件做本地 OpenXML/订单结构识别。只有至少包含一个有效订单工作簿的邮件才写入订单缓存并触发通知。附件检查采用现有 `isOrderWorkbookContent` 规则，并限制单附件最多 25 MB；不执行宏、公式、嵌入对象或外部链接。

### 4. 本地邮件服务

`LocalMailService` 是主进程内的编排边界，向 IPC 暴露以下能力：

- 登录测试、保存设置、重新登录和退出邮箱。
- 读取最近 7 天的本地订单邮件列表。
- 手动刷新并返回最新缓存视图。
- 订阅本地 `new-messages`、连接状态和缓存刷新事件。
- 下载用户选中的邮件附件并调用现有提取链路。

渲染进程不直接创建 IMAP 连接，不读取数据库，也不接触文件系统中的加密凭据。

### 5. 本地提取

用户勾选邮件并点击提取后：

1. 主进程根据选中 UID 从邮箱下载有效订单 Excel 附件到带时间戳的本地临时目录。
2. 调用现有 Python extraction bridge 和 `extract.py`。
3. 继续生成现有 CSV、XLSX 和审计输出，保留当前进度事件与“打开输出目录”按钮。
4. 全部选中邮件处理成功后，将相应 UID 标记为已提取。
5. 某封邮件或附件失败时记录可读错误，不把该 UID 标成成功，允许用户重新勾选。

下载附件、提取结果和审计文件继续保留在现有带时间戳的本地邮件输出目录中，不改变用户当前查找和打开结果的方式。

### 6. 托盘与窗口生命周期

- 使用 `app.setLoginItemSettings` 配置 Windows 用户登录后自动启动。
- 自动启动时允许直接进入托盘，不强制弹出主窗口。
- 点击窗口关闭按钮只隐藏窗口并保持后台监听。
- 托盘菜单提供“打开主界面”“重新连接邮箱”“退出”。
- 点击新邮件通知时恢复并聚焦主窗口。
- 屏幕熄灭不影响运行；系统睡眠期间网络暂停，唤醒后立即补扫。

## 数据流

### 登录

渲染进程提交邮箱地址和授权码 → 主进程建立一次 IMAP TLS 连接验证 → 成功后加密授权码并保存非敏感设置 → 启动监控器 → 初次同步最近 7 天 → 渲染进程读取缓存。

### 新邮件

IMAP IDLE/兜底扫描发现新 UID → 检查候选附件并识别订单工作簿 → SQLite 幂等写入 → 主进程事件推送给渲染进程 → 弹出一次 Windows 通知。

### 手动提取

用户选择 UID → 主进程安全下载附件 → 现有 Python bridge 提取 → 返回本地输出路径和失败明细 → 更新已提取状态 → 界面继续使用现有结果与目录按钮。

## 状态与错误处理

界面和托盘统一使用四种运行状态：

- `stopped`：用户未启用或已退出邮箱。
- `connecting`：正在登录、恢复连接或同步。
- `connected`：IMAP 可用且最近同步成功。
- `attention_required`：授权码失效或需要用户处理。

网络离线时保留并显示本地列表，同时明确标记“离线，显示本地缓存”。恢复连接后补扫。授权错误不循环弹通知。SQLite 重建、同步失败和提取失败均写入脱敏本地日志，并向界面返回可操作的中文错误。

## 删除和保留范围

### 删除或停用

- 桌面端 `RemoteEmailApiClient`、远程 SSE 订阅和远程邮件配置文件。
- 根目录和 `services/orderflow-email-api` 中的邮件 HTTP 服务、broker、Agent、部署、Nginx/systemd 和远程配置脚本。
- `serve:email-api` 及远程服务专用 CI/打包步骤。
- 旧的云网关设计/实施文档应标记为已取代，避免以后误用。

### 保留并复用

- `src/core/emailSource.ts` 的 IMAP、附件筛选与瞬时错误重试能力。
- `src/core/extractionService.ts`、现有 Python bridge、`extract.py` 和输出 writer。
- `src/main/ipcHandlers.ts` 的系统通知、进度事件、打开路径和更新功能。
- 渲染器现有邮件列表、勾选、已提取标记、进度和输出交互。
- 应用更新检查功能；“全部本地”只约束邮件、凭据、缓存与订单数据，不取消版本更新检查。

现有 `codex/office-mail-gateway` 分支不合并。新实现只在 `codex/local-mail-workstation` 分支进行。原主目录中五个 `Deluxe Dry Lining` 未提交修改继续保留，不进入本地邮件功能提交。

## 测试策略

### 单元测试

- DPAPI/safeStorage 注入边界、不可用时不落明文、旧明文设置迁移。
- SQLite WAL、7 天清理、`mailbox_id + uid` 幂等、通知去重和损坏恢复。
- IMAP 初次扫描、UID 增量、IDLE 事件、60 秒兜底、退避、授权暂停、睡眠恢复。
- 订单附件识别、25 MB 限制和仅订单邮件通知。
- 手动勾选提取成功/失败状态和现有输出路径保持不变。
- 窗口关闭到托盘、显式退出和登录自启动。

### 集成与回归

- IPC 只返回脱敏设置；渲染器不能取得授权码。
- 本地邮件列表、刷新、通知、选择和提取端到端通过。
- 现有本地文件提取与 `Deluxe Dry Lining` 规则回归通过。
- 静态检查和运行时冒烟确认桌面进程没有创建入站监听器，也不依赖 `orderflow.ausmet.ai`。
- 完整运行根项目 TypeScript、Vitest、Python 测试和生产构建。
- 生成 Windows 便携版，在真实 Windows 环境验证登录、DPAPI、登录自启动、托盘、系统通知、休眠恢复和提取。

## 完成标准

- 新安装只要求企业邮箱地址和授权码，不要求服务器地址或 token。
- 授权码在 Windows 上仅以 DPAPI 加密形式持久化。
- Windows 登录后程序自动在托盘运行，窗口关闭后仍能发现并通知新订单邮件。
- 固定显示最近 7 天订单邮件，本地缓存可在短暂离线时立即打开。
- 新订单邮件只通知一次，普通邮件不通知。
- 只有手动勾选并点击后才执行提取，结果和输出按钮与当前本地流程一致。
- 应用不开放入站端口，邮件、附件、缓存和提取结果不发送到远程服务。
- 旧云网关分支未合并，五个现有提取修复未被本功能提交污染。
